## Elasticsearch原理详解

**1. 核心原理与基础架构**

Elasticsearch是搜索和分析引擎，本质上是一个**分布式内存数据库**，特别适合构建高效搜索系统。其核心原理基于**倒排索引**技术，这使得它能够实现快速的全文搜索。

**1.1 倒排索引机制**

- **构建过程**：当ES写入文档时，会对需要搜索的字段进行分词，然后以分词为索引组成查找树
- **工作原理**：将文档中的词条映射到包含该词条的文档列表，将全文匹配转换为对树的查找

> *示例*：当写入商品记录时，ES对商品标题进行分词，然后按单词建立索引，形成倒排索引表。

**2. 核心概念**

**2.1 索引(Index)**

- 类似于数据库中的表，用于组织和存储数据，每个索引包含多个分片

**2.2 文档(Document)**

- 索引中的基本数据单元，以JSON格式存储
- 每个文档都有唯一ID

**2.3 映射(Mapping)**

- 定义文档的结构和数据类型
- 指定每个字段的名称、类型、存储方式等属性

**2.4 分片(Shard)**

- 索引的物理分块，数据被均匀分布在不同分片中
- **主分片(Primary Shard)**：提升系统存储容量
- **副本分片(Replica Shard)**：提高数据可用性

> *文档存储算法*：`shard=hash(_routing)%numbers_of_primary_shards`，确保文档均匀分布

**3.2 故障转移机制**

- 当节点故障时，系统自动将主分片迁移到其他节点
- 例如：3个主分片与1个副本分片分布在3个节点上，当一个节点故障后，其他节点会自动接管

**3.3 分布式查询**

- 查询时，协调节点会向所有相关分片发送查询请求
- 收集各分片结果后进行全局排序和聚合

**5. 高级功能**

**5.1 跨字段查询(cross-fields)**

- **术语中心查询**：允许术语出现在任意字段中
- **工作原理**：分析查询字符串生成术语列表，搜索每个术语在任意字段中的出现情况
- **适用场景**：人名搜索、产品搜索等多字段匹配场景

**5.2 聚合查询**

- 对数据进行分组及计算，如分组、计算平均值、最大值、最小值及总和
- 用于数据分析和深度洞察，是ES强大的分析功能

**6. 为什么ES适合搜索场景**

1. **倒排索引**：专为搜索而设计，提供高效全文匹配
2. **分布式架构**：支持水平扩展，处理海量数据
3. **近实时搜索**：数据索引后几秒内即可被搜索到
4. **丰富查询功能**：支持复杂查询、聚合分析
5. **高可用性**：通过副本分片实现数据冗余和故障转移



### 具体案例

假设我们有一个高并发的电商系统，用户在高峰期访问量激增，导致系统频繁发生 Full GC，严重影响了系统的性能和响应时间。

#### 诊断步骤

1. **启用 GC 日志**：
   ```sh
   -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/var/log/jvm_gc.log
   ```

2. **分析 GC 日志**：
   - 使用 GCViewer 分析日志，发现 Eden 区和 Survivor 区内存使用率较高，Old 区内存增长较快。

3. **生成堆转储文件**：
   
   ```sh
   -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/heapdump.hprof
   ```
   - 使用 MAT 分析堆转储文件，发现大量临时对象和大对象。

#### 调优步骤

1. **调整堆内存大小**：
   ```sh
   -Xms4g -Xmx8g
   ```

2. **选择合适的垃圾回收器**：
   ```sh
   -XX:+UseG1GC
   ```

3. **调整 G1 GC 参数**：
   ```sh
   -XX:MaxGCPauseMillis=200
   -XX:InitiatingHeapOccupancyPercent=45
   -XX:NewRatio=3
   -XX:SurvivorRatio=8
   -XX:PretenureSizeThreshold=3145728
   ```

4. **优化代码**：
   - 减少临时对象的创建，使用对象池技术。
   - 优化大对象的使用，避免频繁创建和销毁。

5. **持续监控**：
   - 使用 Prometheus + Grafana 实时监控 JVM 指标，确保系统稳定运行。

### 总结

通过以上步骤，我们可以有效地解决 Full GC 问题，并提高系统的性能和稳定性。JVM 调优是一个持续的过程，需要不断监控和调整，以适应应用的变化和需求。希望这些方法能对你在实际项目中的 JVM 调优提供帮助。

**优点**：

- **灵活性**：可以动态地选择和切换不同的算法。

### 消息积压和消息逆序

4.1 **消息积压**

消息积压通常是由于消费者处理能力不足或网络问题导致的。以下是一些解决方案：

- **增加消费者实例**：通过增加消费者实例来提高消费能力。
- **批量消费**：启用批量消费，一次处理多条消息，提高吞吐量。
- **调整消息队列数**：增加消息队列的数量，分散消费压力。

4.2 **消息逆序**

消息逆序通常是由于多个消费者实例并行消费导致的。以下是一些解决方案：

- **有序消费**：使用 RocketMQ 的有序消费特性，确保消息按照发送顺序被消费。
  
- **单线程消费**：确保每个队列只有一个消费者实例，避免并行消费导致的逆序问题。
- **消息排序**：在消费者端对消息进行排序，确保按顺序处理。

### **1. 持久化机制**
**回答结构**：存储设计 + 刷盘策略 + 高可用保障  
**参考答案**：  
RocketMQ的持久化通过**CommitLog + ConsumeQueue**实现：  

- **CommitLog**：所有消息按顺序写入日志文件（类似Kafka的Partition），保证高吞吐。  
- **ConsumeQueue**：消息的逻辑队列，存储消息在CommitLog中的物理偏移量，实现快速检索。  
- **刷盘策略**：  
  - **同步刷盘**（FlushDiskType=SYNC_FLUSH）：消息写入物理磁盘后才返回ACK（应答信号），保证数据不丢失，但性能较低（适合金融级场景）。  
  - **异步刷盘**（默认，FlushDiskType=ASYNC_FLUSH）：消息先写入PageCache后返回，由后台线程刷盘，吞吐高但极端情况可能丢消息。  
- **高可用**：通过Broker主从同步（同步/异步复制） + DLedger选主，保障数据可靠性。  

---

### **2. 消息可靠性**
**回答结构**：生产者 → Broker → 消费者全链路保障  
**参考答案**：  
RocketMQ通过**全链路ACK机制**保障可靠性：  

1. **生产者端**：  
   - 同步发送(等待broker的确认相应) + 重试机制（默认2次），确保消息到达Broker。   
2. **Broker端**：  
   - 主从同步复制：消息写入主节点后同步到从节点，主宕机时从节点可切换。  
   - 消息存储冗余：CommitLog多副本（需结合DLedger实现）。  
3. **消费者端**：  
   - **ACK机制**：消费者消费成功后返回`CONSUME_SUCCESS`，失败则返回`RECONSUME_LATER`触发重试（默认最多16次）。  
   - **消费位点持久化**：消费者Offset定期提交到Broker，避免重复消费。  

---

### **3. 延迟消息（延迟消费）**
**回答结构**：实现原理 + 使用限制 + 适用场景  
**参考答案**：  
RocketMQ的延迟消息通过**预定义延迟级别**实现（不支持任意时间精度）：  
- **原理**：  
  1. 发送延迟消息时指定`delayTimeLevel`（1~18，对应1s/5s/10s/30s/1m…2h）。  
  2. Broker将消息暂存到`SCHEDULE_TOPIC_XXX`队列，由定时任务扫描并投递到目标Topic。  
- **关键配置**：  
  - 延迟级别在`broker.conf`中通过`messageDelayLevel`自定义。  
  - 定时任务由`ScheduleMessageService`调度，精度受Broker负载影响。  
- **使用场景**：  
  - 订单超时关闭、异步任务延迟触发等。  
  - **注意**：延迟消息在投递前会占用存储空间，需合理设置Topic的TTL。

### Redis

**简述全量同步**和增量同步区别？

- 全量同步：master将完整内存数据生成RDB，发送RDB到slave。后续命令则记录在repl_baklog，逐个发送给slave。
- 增量同步：slave提交自己的offset到master，master获取repl_baklog中从offset之后的命令给slave

**面试题**：**Redis如何判断KEY是否过期呢？**

**答**：在Redis中会有两个Dict，也就是HashTable，其中一个记录KEY-VALUE键值对，另一个记录KEY和过期时间。要判断一个KEY是否过期，只需要到记录过期时间的Dict中根据KEY查询即可。

**面试题**：**Redis何时删除过期KEY？如何删除？**

**答**：Redis的过期KEY处理有两种策略，分别是惰性删除和周期删除。

**惰性删除**是指在每次用户访问某个KEY时，判断KEY的过期时间：如果过期则删除；如果未过期则忽略。

**周期删除**有两种模式：

- **SLOW**模式：通过一个定时任务，定期的抽样部分带有TTL的KEY，判断其是否过期。默认情况下定时任务的执行频率是每秒10次，但每次执行不能超过25毫秒。如果执行抽样后发现时间还有剩余，并且过期KEY的比例较高，则会多次抽样。
- **FAST**模式：在Redis每次处理NIO事件（NIO（Non-blocking I/O，非阻塞I/O）
- 2）之前，都会抽样部分带有TTL的KEY，判断是否过期，因此执行频率较高。但是每次执行时长不能超过1ms，如果时间充足并且过期KEY比例过高，也会多次抽样

**面试题**：**当Redis内存不足时会怎么做**？

**答**：这取决于配置的内存淘汰策略，Redis支持很多种内存淘汰策略，例如LRU、LFU、Random. 但默认的策略是直接拒绝新的写入请求。而如果设置了其它策略，则会在每次执行命令后判断占用内存是否达到阈值。如果达到阈值则会基于配置的淘汰策略尝试进行内存淘汰，直到占用内存小于阈值为止。

多集群多索引
集群 = 一组 Elasticsearch 服务器，负责“存和查”
索引 = 这组服务器里的“某一类数据集合”，有点像 MySQL 里的“表”
所以：

多集群：不是只有一套 ES，而是有多套 ES 环境
多索引：同一套 ES 里，不是只存一种数据，而是分成多个索引分别存

# 22. token 机制的鉴权有什么好处？

# 27. 缓存与数据一致性，你有什么样的理解吗？

辩证两个情况

1、直接和缓存做交互，完全信任缓存

2、和缓存做交互，如果缓存没有，则去和数据库查

![image-20241111164946097](C:\Users\21162\AppData\Roaming\Typora\typora-user-images\image-20241111164946097.png)

根据以上的流程没有问题，但是当数据变更的时候，如何把缓存变到最新，使我们下面要讨论的问题。

#### 1、更新了数据库，再更新缓存

假设数据库更新成功，缓存更新失败，在缓存失效和过期的之前，读取到的都是老数据缓存。

#### 2、更新缓存，更新数据库

缓存更新成功了，数据库更新失败，是不是读取的缓存的都是错误的。

以上两种，全都不推荐。

#### 3、先删除缓存，再更新数据库

有一定的使用量。即使数据库更新失败。缓存也可以会刷。

存在的问题是什么？

高并发情况下！！

比如说有两个线程，一个是 A 线程，一个是 B 线程。

A 线程把数据删了，正在更新数据库，这个时候 B 线程来了，发现缓存没了，又查数据，又放入缓存。缓存里面存的就一直是老数据了。

**延迟双删**。更新完数据库之后，我再删一次。

具体来说，缓存延迟双删的操作流程如下：

1. 在修改数据库中的数据之前，首先删除一次缓存，这是为了确保在数据库数据修改和缓存数据被删除的间隔时间内，如果有任何请求命中该缓存，也不会得到旧的数据。
2. 接下来，执行对数据库的修改操作。
3. 修改完数据库后，不是立即删除缓存，而是等待一段时间后再删除缓存。这个“等待一段时间”的过程就是“延迟”的来源。延迟的目的是为了确保在修改数据库到再次删除缓存的这段时间内，其他事务或请求可能已经基于旧的缓存数据进行了操作，并将这些旧数据更新到了缓存中。如果立即删除缓存，可能会导致这些新更新的旧数据被错误地保留在缓存中，造成数据不一致。

#### 扩展思路

1、消息队列补偿

删除失败的缓存，作为消息打入 mq，mq 消费者进行监听，再次进行重试刷缓存。

2、canal

监听数据库的变化，做一个公共服务，专门来对接缓存刷新。优点业务解耦，业务太多冗余代码复杂度

# 42. 项目有用到定时任务吗？怎么做的？

有的，项目上的定时任务，我们采用的是 xxljob，分布式任务调度，因为我们这一块用的是微服务，必须要考虑多个服务的执行的问题，如果用传统的定时任务方式，防止不了同时执行的问题，可能还有自己写分布式锁来进行排斥处理。不如直接使用 xxljob，xxljob 的调度十分完善，故障转移，负载均衡算法选择都非常方便，引入后，直接配置即可执行。像项目里面同步点赞和收藏的数据，我们就是采用定时任务来进行操作的。定时任务扫描到待同步的数据，扫描后，同步到数据库。

xxljob：

- 1、简单：支持通过 Web 页面对任务进行 CRUD 操作，操作简单，一分钟上手；
- 2、动态：支持动态修改任务状态、启动 / 停止任务，以及终止运行中任务，即时生效；
- 3、调度中心 HA（中心式）：调度采用中心式设计，“调度中心” 自研调度组件并支持集群部署，可保证调度中心 HA；
- 4、执行器 HA（分布式）：任务分布式执行，任务 "执行器" 支持集群部署，可保证任务执行 HA；
- 5、注册中心：执行器会周期性自动注册任务，调度中心将会自动发现注册的任务并触发执行。同时，也支持手动录入执行器地址；
- 6、弹性扩容缩容：一旦有新执行器机器上线或者下线，下次调度时将会重新分配任务；
- 7、路由策略：执行器集群部署时提供丰富的路由策略，包括：第一个、最后一个、轮询、随机、一致性 HASH、最不经常使用、最近最久未使用、故障转移、忙碌转移等；

## 背景

项目中发帖，评论可能包含多个关键词，铭感词。此时需要对该内容进行过滤处理。此前都是在客户端层面操作，这样不仅带来了性能的损耗，而且新增铭感词时，需要客户端重新打包上架，显得十分不合理。所以应该在服务端层面进行算法数据处理。

## DFA 算法

DFA 全称为: Deterministic Finite Automaton, 即确定有穷自动机。其特征为：有一个有效状态的集合和一些从一个状态通向另一个状态的边，每条边上标记有一个符号，其中一个状态是初态，某些状态是终态。但不同于不确定的有限自动机，DFA 中不会有从同一个状态触发的两条边标志有相同的符号。



确定：状态以及引起状态转换的事件都是可确定的，不存在“意外”

有穷： 状态以及事件的数量都是可穷举的

## websocket

接受消息



1.1 产生背景及原因：

WebSocket 协议在2008年诞生，2011年成为国际标准。所有浏览器都已经支持了

举例来说，我们想了解今天的天气，只能是客户端向服务器发出请求，服务器返回查询结果。HTTP 协议做不到服务器主动向客户端推送信息。

为什么需要webSocket

HTTP 协议有一个缺陷：通信只能由客户端发起。

1.2 简述

WebSocket 是基于 TCP 的一种新的网络协议。它实现了浏览器与服务器全双工通信——浏览器和服务器只需要完成一次握手，两者之间就可以创建持久性的连接， 并进行双向数据传输。

它的最大特点就是，服务器可以主动向客户端推送信息，客户端也可以主动向服务器发送信息，是真正的双向平等对话，属于服务器推送技术的一种。

1.3 HTTP协议和WebSocket协议对比：

**全双工通信**：一旦建立连接，客户端和服务器都可以随时向对方发送数据。

**低延迟**：由于减少了握手次数和头部信息量，WebSocket 提供了比传统 HTTP 更低的延迟。

短连接：客户端向服务器发送请求建立连接之后，当服务器响应完之后连接就不存在了

长连接：建立连接之后，连接一直存在

1.5 疑问

既然WebSocket支持双向通信，功能看似比HTTP强大，那么我们是不是可以基于WebSocket开发所有的业务功能？

答案并不是。

WebSocket缺点：

• 服务器长期维护长连接需要一定的成本

• 各个浏览器支持程度不一

• WebSocket 是长连接，受网络限制比较大，需要处理好重连

结论：WebSocket并不能完全取代HTTP，它只适合在特定的场景下使用
